# 改进计划:让 agent 更聪明,但不放松治理

记录于 2026-07-20 架构讨论。背景:目前 agent 只能回答"能渲染成一张图"的问题,
数据相关的宽泛问题(能问什么 / 哪个更高 / 为什么)全部死路。诊断结论:不是治理
太严,而是 **工具面太窄 + 模型看不到数据 + 提示词没教组合**。修法全部在治理边界
内——LLM 依然不写 SQL、不产 ViewSpec、结论只走 verdict 瓦片。

## 明天做:第一批(①+②+③,一个连贯切片)✅ 已完成 2026-07-20

实现记录:① `describeDataSource`(semantic-model.ts)+ `describeData` 工具 + chat.tsx
瓦片/进度注册;② `summarizeSpec` 拆成 base + `decisionDetail`,4KB 上限超限回退;
③ 系统提示两条新规则(目录不拒绝、判断题先渲染证据再 verdict)+ 多图一 verdict
的脚本化 agent 测试。

### ① `describeData` 目录工具 —— 回答"你有什么数据 / 能问什么"

- 新工具,加进 `src/agent/tools.ts` 的 `analysisTools`,并在 `trigger/house-agent.ts`
  的 `chat.agent({ tools })` 声明(不变式 3:必须声明在 agent 上,否则 turn 2 起
  toModelOutput 失效)。
- 无参数(或可选 sourceId)。从语义模型注册表读:数据源、行数规模、时间范围
  (`availableRange`/`lastRefresh`)、每个度量(label + description + aggregation)、
  每个维度(label + kind + 值域规模/示例值)。
- 输出一张 **table 瓦片**(列:名称/类型/说明/示例),不跑 SQL,纯语义层。
- `toModelOutput` 压缩成一行("catalog shown: N measures, M dimensions")。
- 系统提示:用户问"有什么数据/能问什么/数据从哪来"时调它,别拒绝。

### ② 选择性放宽"决策数据" —— 模型能对结果做推理

现状:`toModelOutput` 把渲染结果压成一行,模型无法回答"有几个区超过 50 万"这类
图里明明有答案的问题。改法(改 `src/analysis/pipeline.ts` 的 `summarizeSpec` 或
render 工具的 `toModelOutput`):

- comparison(≤40 行):附完整 `label=value` 列表(~1–2KB,可控)。
- timeseries/area:每条系列的首值、末值、最大、最小及其时间点。
- pie:全部切片 label + 份额。
- table:行数 + 前几行。
- kpi/distribution:已足够。
- 约束:总量必须有上限(比如 4KB),超限退回现在的一行摘要——决策数据放宽是
  **有界的**,不是把渲染数据整个塞回 prompt(不变式 2 依然成立)。

### ③ 教会组合 —— 判断类问题 = 多图 + verdict 下结论

- 系统提示加一条:"当问题需要判断(哪个好/贵不贵/该选哪个),先调用一次或多次
  renderAnalysis 渲染支撑证据,再用 emitVerdict 给出基于渲染摘要事实的结论。"
- 循环本身早就支持(stepCountIs 预算 15 步),纯提示词改动。
- 配合 ②,verdict 才有真数据可引用(verdict 事实锚定已在 commit 0bd6396 做了,
  verdict 只许陈述 render 摘要里出现的事实——②扩大的正是这个摘要)。
- 测试:脚本化 agent 测试,一轮两次 renderAnalysis + 一次 verdict 的流程断言。
